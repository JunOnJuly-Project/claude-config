# Clean Architecture

의존성 규칙을 준수하는 동심원 레이어 구조로 설계합니다.

## 입력
`$ARGUMENTS` — 설계/구현할 기능 또는 리팩토링 대상

## 핵심 원칙: 의존성 규칙
**바깥 레이어가 안쪽 레이어에만 의존한다. 안쪽은 바깥을 모른다.**

```
┌─────────────────────────────────────────┐
│  Frameworks & Drivers (프레임워크/드라이버) │
│  ┌─────────────────────────────────────┐ │
│  │  Interface Adapters (인터페이스 어댑터) │ │
│  │  ┌─────────────────────────────────┐ │ │
│  │  │  Application (유스케이스)          │ │ │
│  │  │  ┌─────────────────────────────┐ │ │ │
│  │  │  │  Entities (엔티티)            │ │ │ │
│  │  │  └─────────────────────────────┘ │ │ │
│  │  └─────────────────────────────────┘ │ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

## 레이어 구조

### 1. Entities (엔티티 레이어)
- 핵심 비즈니스 규칙
- 어떤 외부 의존성도 없음
- 가장 변경 가능성이 낮은 코드

### 2. Use Cases (유스케이스 레이어)
- 애플리케이션 비즈니스 규칙
- 엔티티 레이어에만 의존
- 입력/출력 포트(인터페이스) 정의

### 3. Interface Adapters (인터페이스 어댑터)
- 유스케이스와 외부 세계 사이의 변환
- Controller, Presenter, Gateway
- DTO ↔ 도메인 모델 변환

### 4. Frameworks & Drivers (프레임워크/드라이버)
- 웹 프레임워크, DB, 외부 API
- 가장 바깥 레이어, 가장 세부적
- 구현 세부사항

## 디렉토리 구조
```
src/
├── entities/                # 엔티티 레이어
│   ├── {entity}.ts          # 엔티티 + 비즈니스 규칙
│   └── errors.ts            # 도메인 에러
├── use-cases/               # 유스케이스 레이어
│   ├── {use-case}.ts        # 유스케이스 구현
│   ├── ports/
│   │   ├── input.ts         # 입력 포트 (인터페이스)
│   │   └── output.ts        # 출력 포트 (인터페이스)
│   └── dto/                 # 유스케이스 DTO
├── adapters/                # 인터페이스 어댑터
│   ├── controllers/         # HTTP 컨트롤러
│   ├── presenters/          # 응답 포맷터
│   └── gateways/            # DB/외부 API 게이트웨이
└── frameworks/              # 프레임워크/드라이버
    ├── web/                 # 웹 프레임워크 설정
    ├── db/                  # DB 설정/마이그레이션
    └── config/              # 앱 설정
```

## 의존성 주입
- 유스케이스는 출력 포트(인터페이스)에 의존
- 게이트웨이가 출력 포트를 구현
- 컴포지션 루트에서 의존성 조립

## 독립 사용 시
- `/clean-arch 설계` — 현재 프로젝트에 Clean Architecture 구조 제안
- `/clean-arch 리팩토링` — 기존 코드를 Clean Architecture로 전환

## `/develop`과 조합 시
- Phase 1에서 레이어 구조 설계
- Phase 4에서 의존성 방향 준수하며 구현
