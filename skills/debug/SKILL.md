# 체계적 디버깅

과학적 방법론에 따라 체계적으로 버그를 추적하고 수정합니다.

## 입력
`$ARGUMENTS` — 에러 설명, 스택 트레이스, 재현 방법, 또는 이슈 번호

## 실행 단계

### 1단계: 재현 (Reproduce)
- 보고된 증상/에러 확인
- 에러 메시지, 스택 트레이스 수집
- 최소 재현 시나리오 구성
- 재현 가능 여부 확인 → 재현 불가 시 추가 정보 요청

### 2단계: 분석 (Analyze)
- 스택 트레이스에서 관련 소스 파일 읽기
- 실행 경로 추적 (함수 호출 체인)
- 로그 분석
- 관련 의존성/설정 확인

### 3단계: 가설 수립 (Hypothesize)
```markdown
## 가설 목록
1. {가설 A} — 가능성: 높음/중간/낮음
   근거: {왜 이 가설이 유력한지}
2. {가설 B} — 가능성: ...
```
- 가장 유력한 가설부터 검증

### 4단계: 수정 (Fix)
- 최소한의 코드 변경으로 수정
- 클린코드 원칙 준수
- 부작용 확인

### 5단계: 회귀 테스트 (Regression Test)
- 이 버그를 잡는 테스트 작성
- 테스트명: `test_이슈번호_버그설명` 또는 `it('이슈번호 - 버그설명')`
- 전체 테스트 스위트 실행 → 기존 테스트 통과 확인

### 6단계: 커밋 및 보고
커밋: `fix({도메인}): {버그 설명} (Closes #{이슈번호})`

보고서:
```markdown
# 디버깅 보고서

## 증상
{보고된 문제}

## 근본 원인
{발견된 원인과 영향 범위}

## 수정 내용
{적용한 변경사항, 수정 파일 목록}

## 회귀 테스트
{추가한 테스트와 커버하는 시나리오}

## 교훈
{향후 방지 방안}
```

## 출력
- 모든 분석 및 보고 **한국어** 작성
- 추측하지 않고, 검증된 사실만 보고
