# 자율 개발 워크플로우

사용자의 개발 지시를 받아 전체 개발 사이클을 자율적으로 수행합니다.

## 입력
`$ARGUMENTS` — `[방법론 플래그] 기능 설명`

### 방법론 플래그 (선택, 중복 가능)
- `--tdd` — Test-Driven Development (Red→Green→Refactor)
- `--ddd` — Domain-Driven Design (바운디드 컨텍스트, 엔티티, VO)
- `--clean-arch` — Clean Architecture (동심원 레이어)
- `--hexagonal` — Hexagonal Architecture (포트/어댑터)
- `--event-driven` — Event-Driven Architecture
- `--cqrs` — CQRS 패턴
- `--bdd` — Behavior-Driven Development (Given/When/Then)
- `--fp` — Functional Programming 패턴
- `--microservice` — Microservice 패턴

예시:
- `/develop --tdd --ddd 사용자 인증 기능`
- `/develop --clean-arch REST API 서버`
- `/develop 간단한 유틸리티 함수` (방법론 없이)

## 계획서 연동 (`/start`에서 호출 시)
`/start`에서 호출되면 `docs/plans/{프로젝트명}-plan.md` 계획서가 이미 존재합니다.
이 경우 Phase 1의 계획 수립을 건너뛰고 계획서를 참조하여 실행합니다:
- 계획서의 작업 분해에서 현재 태스크의 세부 사항 확인
- 계획서의 이슈 번호를 커밋에 참조 (`Refs #{이슈번호}`)
- 계획서의 디렉토리 구조, 기술 스택 결정 준수
- 계획서의 테스팅 전략 참조

## 워크플로우

### Phase 1: 계획
1. `$ARGUMENTS`에서 방법론 플래그와 기능 설명을 파싱한다
2. 현재 프로젝트 구조를 분석한다 (Glob, Read)
3. 도메인을 식별하고 작업을 세부 태스크로 분해한다
4. GitHub 이슈를 도메인별로 생성한다 (gh CLI):
   - 제목: `[{도메인}] {기능 설명}`
   - 라벨: `domain:{도메인}`, `type:feature`, `priority:{우선순위}`
   - 본문: 요구사항, 수용 기준, 작업 체크리스트
5. Git Flow에 따라 feature 브랜치를 생성한다:
   `git checkout -b feature/{도메인}/{기능} develop`
6. `docs/plans/`에 계획 문서를 작성한다
7. 방법론별 추가 작업:
   - `--ddd`: 바운디드 컨텍스트, 도메인 모델 설계
   - `--clean-arch`: 레이어 구조 설계
   - `--hexagonal`: 포트/어댑터 식별
   - `--microservice`: 서비스 경계 정의

### Phase 2: 문서화
1. API 인터페이스를 설계하고 문서화한다
2. 커밋: `docs({도메인}): {인터페이스/문서} 정의 (Refs #{이슈})`
3. 방법론별 추가:
   - `--event-driven`: 이벤트 흐름 문서
   - `--cqrs`: 커맨드/쿼리 분리 문서
   - `--ddd`: 도메인 모델 문서

### Phase 3: 테스트
**`--tdd` 또는 `--bdd` 플래그 시:**
1. 실패하는 테스트를 먼저 작성한다
2. 테스트를 실행하여 모두 실패하는지 확인한다
3. 커밋: `test({도메인}): {테스트 설명} (Refs #{이슈})`
4. `--bdd` 시 Given/When/Then 형식 사용

**플래그 없을 시:**
- Phase 4 구현 후에 테스트를 작성한다

### Phase 4: 구현
1. 최소 구현 단위별로 코드를 작성한다
2. 클린코드 원칙을 준수한다:
   - 함수 20줄 이내, 매개변수 3개 이내
   - 한국어 주석 (WHY만)
   - SOLID, DRY, KISS
3. 각 함수/모듈 구현마다 개별 커밋:
   `feat({도메인}): {기능 설명} (Refs #{이슈})`
4. TDD 시: 테스트를 통과시키는 최소 코드 작성
5. 방법론별 추가:
   - `--fp`: 순수 함수, 불변성, 합성
   - `--ddd`: 레이어 분리, 엔티티/VO/리포지토리
   - `--clean-arch`: 의존성 방향 준수

### Phase 5: 리팩토링
1. 테스트가 모두 통과한 상태에서 코드를 개선한다
2. DRY 위반, 네이밍, 복잡도를 개선한다
3. 리팩토링 후 테스트 재확인
4. 커밋: `refactor({도���인}): {개선 내용} (Refs #{이슈})`

### Phase 6: 자체 리뷰
1. 변경된 모든 파일을 검토한다
2. 클린코드 체크리스트 적용:
   - [ ] SOLID 준수
   - [ ] DRY — 중복 없음
   - [ ] 함수 길이/매개변수
   - [ ] 네이밍 적절성
   - [ ] 한국어 주석
   - [ ] 보안 취약점 없음
   - [ ] 성능 문제 없음
3. 발견된 문제 수정 후 개별 커밋

### Phase 7: 문서 업데이트
1. README.md 업데이트 (필요 시)
2. CHANGELOG.md 업데이트
3. API 문서 최종 업데이트
4. 커밋: `docs({도메인}): {문서} 갱신 (Refs #{이슈})`

### Phase 8: 완료 보고
사용자에게 한국어로 보고:
```markdown
## 개발 완료 보고

### 구현된 기능
{기능 요약}

### 적용된 방법론
{사용한 방법론과 적용 내용}

### 생성/수정된 파일
{파일 목록}

### 커밋 히스토리
{커밋 목록}

### 테스트 결과
- 통과: {n}개 | 실패: {n}개
- 커버리지: {n}% (가능 시)

### 브랜치 상태
- 현재 브랜치: feature/{도메인}/{기능}
- develop 머지 대기

### 다음 단계
{PR 생성, 추가 기능 등 제안}
```

## 오류 처리
- 테스트 실패: 원인 분석 후 수정. 3회 시도 실패 시 사용자 보고
- 빌드 실패: 에러 로그 분석, 의존성 문제 해결 시도
- 불확실한 결정: 사용자에게 선택지 제시 후 결정 요청

## 주의
- 모든 출력 **한국어**
- Phase 끝에 일괄 커밋하지 않음 — 최소 단위마다 즉시 커밋
- 사용자 개입 없이 Phase 1~8 연속 수행 (모호함/치명적 오류 시에만 질문)
