# Hexagonal Architecture (포트와 어댑터)

도메인을 중심에 두고 외부 의존성을 포트/어댑터로 분리합니다.

## 입력
`$ARGUMENTS` — 설계/구현할 기능 또는 리팩토링 대상

## 핵심 개념

### 포트 (Port)
- 도메인이 외부와 소통하는 **인터페이스**
- **인바운드 포트 (Driving Port)**: 외부 → 도메인 (유스케이스 인터페이스)
- **아웃바운드 포트 (Driven Port)**: 도메인 → 외부 (리포지토리, 외부 서비스 인터페이스)

### 어댑터 (Adapter)
- 포트의 **구현체**
- **인바운드 어댑터 (Driving Adapter)**: REST 컨트롤러, CLI, 메시지 핸들러
- **아웃바운드 어댑터 (Driven Adapter)**: DB 리포지토리, HTTP 클라이언트, 메일 발송

### 구조 다이어그램
```
 [REST API]     [CLI]     [메시지큐]
     │            │            │
     └────────────┼────────────┘
          인바운드 어댑터
                  │
          ┌───────▼───────┐
          │  인바운드 포트   │
          │  (유스케이스)    │
          │                │
          │   도메인 모델    │
          │  (엔티티, VO)   │
          │                │
          │  아웃바운드 포트  │
          └───────┬───────┘
                  │
          아웃바운드 어댑터
                  │
     ┌────────────┼────────────┐
     │            │            │
  [DB]       [외부 API]    [파일]
```

## 디렉토리 구조
```
src/
├── domain/                  # 도메인 (중심)
│   ├── model/               # 엔티티, VO
│   └── service/             # 도메인 서비스
├── application/             # 애플리케이션
│   ├── port/
│   │   ├── in/              # 인바운드 포트 (유스케이스 인터페이스)
│   │   └── out/             # 아웃바운드 포트 (리포지토리 인터페이스)
│   └── service/             # 유스케이스 구현
└── adapter/                 # 어댑터
    ├── in/
    │   ├── web/             # REST 컨트롤러
    │   └── cli/             # CLI 핸들러
    └── out/
        ├── persistence/     # DB 어댑터
        └── external/        # 외부 API 어댑터
```

## 핵심 규칙
1. 도메인은 어떤 어댑터도 알지 못한다
2. 어댑터는 포트를 통해서만 도메인과 소통한다
3. 아웃바운드 포트는 도메인에 정의, 어댑터에서 구현 (DIP)
4. 어댑터는 쉽게 교체 가능해야 한다 (DB 교체, API 교체)

## 테스트 전략
- **단위 테스트**: 도메인 모델, 유스케이스 (어댑터 Mock)
- **통합 테스트**: 어댑터 (실제 DB, 실제 API)
- **E2E 테스트**: 인바운드 어댑터 → 도메인 → 아웃바운드 어댑터

## 독립 사용 시
- `/hexagonal 설계` — Hexagonal 구조 설계 및 제안
- `/hexagonal 리팩토링` — 기존 코드를 포트/어댑터로 전환
