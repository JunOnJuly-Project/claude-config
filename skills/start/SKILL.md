# 프로젝트 시작 오케스트레이터

에이전트 팀을 소집하여 브레인스토밍을 진행하고, 사용자 승인 후 전체 개발 워크플로우를 실행합니다.

## 입력
`$ARGUMENTS` — `[방법론 플래그] 프로젝트/기능 설명`

### 방법론 플래그 (선택, 중복 가능)
`--tdd`, `--ddd`, `--clean-arch`, `--hexagonal`, `--event-driven`, `--cqrs`, `--bdd`, `--fp`, `--microservice`

예시:
- `/start --tdd --ddd 전자상거래 플랫폼`
- `/start --clean-arch --hexagonal REST API 서버`
- `/start 간단한 CLI 도구`

## Phase -1: 정보 수집 및 프롬프트 정제 (인터뷰)

사용자가 개략적인 정보만 제공한 경우, 체계적으로 질문하여 누락된 정보를 수집하고 최적화된 프로젝트 프롬프트를 생성합니다.

### -1-1. 입력 파싱
1. `$ARGUMENTS`에서 `--xxx` 방법론 플래그와 프로젝트 설명을 분리한다
2. 파싱 결과:
   - `METHODOLOGY_FLAGS`: 방법론 플래그 목록
   - `RAW_DESCRIPTION`: 사용자가 제공한 원본 설명

### -1-2. 정보 충분성 판단
사용자의 입력에서 아래 항목의 충족 여부를 분석한다:

| 카테고리 | 필요 정보 | 필수/선택 |
|----------|-----------|-----------|
| **목적** | 프로젝트가 해결할 문제, 핵심 가치 | 필수 |
| **사용자** | 대상 사용자/클라이언트, 사용 환경 | 필수 |
| **핵심 기능** | 주요 기능 목록 (최소 3개) | 필수 |
| **기술 선호** | 언어, 프레임워크, DB 선호사항 | 선택 (없으면 에이전트가 제안) |
| **규모** | 예상 사용자 수, 데이터 규모 | 선택 (없으면 기본 규모 가정) |
| **제약사항** | 기한, 예산, 필수 사용 기술, 호환성 | 선택 |
| **비기능 요구** | 성능, 보안, 가용성 수준 | 선택 |
| **참고** | 유사 서비스, 참고 디자인, 기존 코드 | 선택 |

### -1-3. 기능 제안 (프로젝트 유형 기반)

프로젝트 유형을 파악한 뒤, 기능을 3가지 카테고리로 분류하여 사용자에게 제시한다.
사용자는 체크리스트에서 원하는 기능을 선택/해제할 수 있다.

**기능 카테고리**:

#### 일반 기능 (이 유형의 프로젝트에서 거의 반드시 포함되는 것)
- 미선택 시 경고: "이 기능 없이 진행하면 {이유}의 문제가 있을 수 있습니다"
- 예 (모의 주식투자): 회원가입/로그인, 매수/매도 주문, 포트폴리오 조회, 잔고 관리

#### 추천 기능 (포함하면 완성도가 크게 올라가는 것)
- 왜 추천하는지 이유를 한줄로 설명
- 예 (모의 주식투자):
  - 수익률 랭킹 — "경쟁 요소로 사용자 참여도를 높입니다"
  - 거래 내역/차트 — "투자 패턴 분석에 필수적입니다"
  - 관심 종목 — "UX를 크게 개선합니다"

#### 유니크 기능 (차별화할 수 있는 창의적 기능)
- 유사 서비스에서 잘 안 보이는, 이 프로젝트를 특별하게 만들 수 있는 기능
- 예 (모의 주식투자):
  - AI 투자 조언 — "사용자의 포트폴리오를 분석하여 추천"
  - 모의 투자 대회 — "기간제 대회로 사용자 유입"
  - 소셜 트레이딩 — "다른 사용자의 포트폴리오 구독/복사"
  - 투자 일지 — "매매 근거를 기록하여 학습 효과"

**제시 형식**:
```
이런 유형의 프로젝트에서 일반적으로 포함되는 기능과 추천 기능을 정리했습니다.
원하는 기능에 번호로 답해주세요 (예: "1,2,3,5,8" 또는 "전부" 또는 "기본만").

### 일반 기능 (기본 포함)
1. [v] 회원가입/로그인 — 사용자 인증 필수
2. [v] 매수/매도 주문 — 핵심 거래 기능
3. [v] 포트폴리오 조회 — 보유 종목, 수익률
4. [v] 잔고 관리 — 가상 자금 충전/관리

### 추천 기능 (완성도 향상)
5. [ ] 실시간 시세 조회 — 실제 시장 데이터 연동
6. [ ] 거래 내역/차트 — 투자 패턴 분석에 필수
7. [ ] 수익률 랭킹 — 경쟁 요소로 참여도 향상
8. [ ] 관심 종목 — UX 개선, 빠른 접근

### 유니크 기능 (차별화)
9. [ ] AI 투자 조언 — 포트폴리오 분석 기반 추천
10. [ ] 모의 투자 대회 — 기간제 대회로 재미 요소
11. [ ] 소셜 트레이딩 — 다른 사용자 포트폴리오 구독
12. [ ] 투자 일지 — 매매 근거 기록으로 학습 효과

### 직접 추가
13. [ ] 기타: ___
```

### -1-4. 인터뷰 (반복 질문)

기능 선택 후 나머지 누락 정보를 수집한다.

**질문 규칙**:
- 한 번에 최대 5개 질문
- 카테고리별로 묶어서 질문
- 선택지를 제시하여 답변 용이하게
- "패스" 옵션 포함

**애매한 답변 처리** (핵심):
사용자 응답이 아래에 해당하면 **재질문**한다. 넘어가지 않는다.

| 애매한 답변 유형 | 재질문 방식 |
|-----------------|------------|
| 너무 짧음 ("그냥 기본으로") | 구체적 선택지 제시: "기본이 A, B 중 어떤 의미인가요?" |
| 모순됨 ("간단하게 + 실시간 처리") | 모순 지적 후 우선순위 질문: "간단함과 실시간 중 어느 쪽이 더 중요한가요?" |
| 범위 불명확 ("적당히") | 구체적 수치/선택지 제시: "10명 규모? 100명? 1000명?" |
| 여러 해석 가능 ("좋은 걸로") | 해석 후보를 나열하고 선택 요청 |

**재질문 형식**:
```
답변이 조금 애매해서 확인이 필요합니다:

> 사용자 원문: "{사용자 답변}"

혹시 아래 중 어느 쪽인가요?
A) {해석 1} — {이런 경우 이렇게 진행}
B) {해석 2} — {이런 경우 이렇게 진행}
C) 다른 의미라면 좀 더 설명해주세요
```

**반복 조건**:
- 필수 항목이 명확해질 때까지 반복 (최대 3회)
- 3회 반복 후에도 불명확하면: "제가 {해석}으로 이해하고 진행합니다. 나중에 수정 가능합니다."로 가장 합리적인 해석을 채택

### -1-5. 정보 종합 및 보강
사용자 응답을 원본 설명과 합쳐 정보를 종합한다:
- 사용자가 "패스"한 항목: 합리적 기본값을 채워 넣고 **표시**: `(기본값: {값})`
- 선택된 기능 목록을 일반/추천/유니크로 분류 정리
- 암묵적 요구사항 추출: 사용자가 선택한 기능에서 당연히 필요한 하위 기능 자동 추가
  (예: "매수/매도" 선택 → 잔고 검증, 체결 알림이 암묵적으로 필요)
  (예: "랭킹" 선택 → 수익률 계산 로직이 암묵적으로 필요)
- 자동 추가된 암묵적 요구사항은 사용자에게 보여주고 확인받는다

### -1-6. 최적화된 프로젝트 프롬프트 생성
수집된 정보를 구조화된 프로젝트 프롬프트로 정리한다:

```markdown
## 프로젝트 정의서

### 개요
- **프로젝트명**: {이름}
- **한줄 설명**: {목적과 핵심 가치}
- **대상 사용자**: {누구를 위한 것인지}
- **플랫폼**: {웹/모바일/CLI/데스크톱}

### 기능 목록

#### 일반 기능 (필수)
1. {기능} — {설명}
2. {기능} — {설명}

#### 추천 기능 (선택됨)
3. {기능} — {설명} / {추천 이유}
4. {기능} — {설명} / {추천 이유}

#### 유니크 기능 (선택됨)
5. {기능} — {설명} / {차별화 포인트}

#### 암묵적 요구사항 (자동 추가)
- {기능 X} 선택에 따라 → {하위 기능} 필요
- {기능 Y} 선택에 따라 → {하위 기능} 필요

### 기술 요구사항
- **언어/프레임워크**: {선택 또는 추천}
- **데이터베이스**: {선택 또는 추천}
- **외부 연동**: {API, 서비스}

### 비기능 요구사항
- **규모**: {예상 사용자/데이터}
- **성능**: {응답 시간, 처리량}
- **보안**: {인증, 데이터 보호 수준}

### 제약사항
- {있으면 기재, 없으면 "없음"}

### 적용 방법론
- {METHODOLOGY_FLAGS에서 파싱된 방법론}
```

### -1-7. 프롬프트 승인
정리된 프로젝트 정의서를 사용자에게 제시한다:

```
위 내용으로 프로젝트를 시작합니다. 수정할 사항이 있으면 말씀해주세요.
맞으면 "확인"이라고 답해주세요. → Phase 0 브레인스토밍으로 진행합니다.
```

**사용자 응답 처리**:
- **확인**: `PROJECT_DESCRIPTION`에 정제된 프롬프트를 저장하고 Phase 0으로 진행
- **수정 요청**: 해당 부분 수정 후 재제시
- 수정 요청 시에도 추가 질문이 필요하면 다시 인터뷰

## Phase 0: 브레인스토밍 (에이전트 팀)

Phase -1에서 정제된 `PROJECT_DESCRIPTION`을 사용하여 브레인스토밍을 시작한다.

### 0-1. 환경 분석
1. 현재 디렉토리의 프로젝트 구조를 파악한다 (Glob, Read)
   - package.json, pyproject.toml, tsconfig.json, go.mod 등
   - 기존 소스 코드 패턴, 디렉토리 구조
2. git 초기화 여부 확인 (`git status`)
3. GitHub 리포지토리 연결 여부 확인 (`gh repo view`)
4. 결과를 `CONTEXT_SUMMARY`로 정리

### 0-2. 라운드 1 — 개별 관점 수집 (4개 에이전트 병렬)

다음 4개 에이전트를 **동시에** Agent 도구로 소집한다:

**planner 에이전트** — 아키텍처 초안:
```
프로젝트: {PROJECT_DESCRIPTION}
방법론: {METHODOLOGY_FLAGS}
환경: {CONTEXT_SUMMARY}

[브레인스토밍 모드] 아래 항목의 초안을 작성하라:
1. 도메인 분석 (핵심/보조 도메인, 의존성 관계)
2. 아키텍처 제안 (선택된 방법론 반영)
3. 디렉토리 구조 제안
4. 작업 분해 (도메인별 태스크, 우선순위, 의존성)
5. 기술 스택 제안
```

**developer 에이전트** — 실현 가능성 평가:
```
프로젝트: {PROJECT_DESCRIPTION}
방법론: {METHODOLOGY_FLAGS}
환경: {CONTEXT_SUMMARY}

[브레인스토밍 모드] 아래 항목에 대한 의견을 제시하라:
1. 기술 스택 실현 가능성 평가
2. 구현 복잡도 추정 (도메인/기능별)
3. 잠재적 기술 부채 요소
4. 라이브러리/프레임워크 추천 (근거 포함)
5. 방법론 적용 시 구현 전략
```

**reviewer 에이전트** — 위험 요소 분석:
```
프로젝트: {PROJECT_DESCRIPTION}
방법론: {METHODOLOGY_FLAGS}
환경: {CONTEXT_SUMMARY}

[브레인스토밍 모드] 아래 항목에 대한 의견을 제시하라:
1. 아키텍처 위험 요소 (확장성, 유지보수성)
2. 보안 고려사항 (인증, 입력 검증, 데이터 보호)
3. 성능 병목 가능성
4. 코드 품질 기준 및 리뷰 중점 사항
5. 방법론 적용 시 주의점
```

**tester 에이전트** — 테스팅 전략:
```
프로젝트: {PROJECT_DESCRIPTION}
방법론: {METHODOLOGY_FLAGS}
환경: {CONTEXT_SUMMARY}

[브레인스토밍 모드] 아래 항목에 대한 의견을 제시하라:
1. 테스팅 전략 (단위/통합/E2E 비율)
2. 핵심 테스트 시나리오 목록
3. 테스트 가능성 관점의 설계 제안
4. 테스트 프레임워크 추천
5. 방법론별 테스트 접근법
```

**라운드 1 완료 후**: 모든 결과를 `docs/plans/brainstorm-round1.md`에 기록한다.

### 0-3. 라운드 2 — 교차 검토 및 토론 (4개 에이전트 병렬)

라운드 1의 전체 결과를 각 에이전트에게 전달하여 상호 검토한다:

**planner 에이전트** — 피드백 반영 및 계획 수정:
```
[라운드 1 전체 결과]

다른 에이전트의 피드백을 반영하여 계획을 수정하라:
- Developer의 복잡도/실현 가능성 피드백 반영
- Reviewer의 위험 요소에 대한 대응 방안 추가
- Tester의 테스트 가능성 제안 반영
- 최종 작업 분해를 GitHub 이슈 형식으로 구조화
```

**developer 에이전트** — 구현 순서 최적화:
```
[라운드 1 전체 결과]

Planner의 아키텍처와 작업 분해를 검토하고:
- 구현 순서 최적화 제안
- 누락된 기술적 태스크 보충
- 각 태스크의 예상 커밋 수 추정
```

**reviewer 에이전트** — 최종 위험 점검:
```
[라운드 1 전체 결과]

제안된 계획 전체를 리뷰하고:
- [치명적] 수준 위험 요소 최종 점검
- 아키텍처 결정에 대한 승인/반대 의견
- 놓친 보안/성능 고려사항 보충
```

**tester 에이전트** — 구체적 테스트 계획:
```
[라운드 1 전체 결과]

확정된 아키텍처를 기반으로:
- 구체적 테스트 계획 수립 (모듈별 테스트 목록)
- 테스트 우선순위 결정
- CI/CD 파이프라인 테스트 단계 제안
```

**라운드 2 완료 후**: 결과를 `docs/plans/brainstorm-round2.md`에 기록한다.

### 0-4. 라운드 3 — 최종 통합 (planner 에이전트 단독)

**planner 에이전트**에게 최종 통합을 요청한다:
```
라운드 1, 2의 모든 논의를 반영하여 최종 개발 계획서를 작성하라.

## 필수 포함 항목:
1. 프로젝트 개요
2. 도메인 분석 및 아키텍처 (다이어그램 포함)
3. 디렉토리 구조
4. 기술 스택 (선택 근거 포함)
5. 작업 분해 (GitHub 이슈 형식):
   각 이슈마다:
   - 제목: [{도메인}] {기능 설명}
   - 라벨: domain:{도메인}, type:{유형}, priority:{우선순위}
   - 수용 기준 (체크리스트)
   - 세부 작업 목록
   - 의존하는 이슈 (있으면)
6. 테스팅 전략
7. 위험 요소 및 대응 방안
8. 적용 방법론 가이드

## 출력:
`docs/plans/{프로젝트명}-plan.md`에 저장
```

### 0-5. 사용자 승인 요청

최종 계획서를 사용자에게 요약 제시하고 승인을 요청한다:

```markdown
## 브레인스토밍 완료

### 참여 에이전트
- Planner (Opus): 아키텍처 설계, 작업 분해
- Developer (Sonnet): 실현 가능성, 기술 스택
- Reviewer (Opus): 위험 요소, 보안, 품질
- Tester (Sonnet): 테스팅 전략, 시나리오

### 계획 요약
{핵심 내용 3~5줄}

### 주요 결정사항
{기술 스택, 아키텍처 패턴, 방법론 적용 방식}

### 식별된 위험 요소
{주요 위험과 대응 방안}

### 예상 이슈: {n}개 이슈, {n}개 도메인

전체 계획서: `docs/plans/{프로젝트명}-plan.md`

**선택해주세요:**
1. 승인 — 계획대로 진행
2. 수정 요청 — 특정 부분 변경
3. 거부 — 브레인스토밍 재시작
```

**사용자 응답 처리**:
- **승인**: Phase 1로 진행
- **수정 요청**: planner에게 수정 전달 → 계획 갱신 → 재제시
- **거부**: 사용자의 추가 지시사항을 반영하여 라운드 1부터 재시작

## Phase 1: Git 환경 설정 (자동, 블로킹 금지)

**원칙**: 사용자에게 묻지 말고 자동으로 git을 준비한다. 저장소가 없으면 무조건 `git init` 부터 한다.

1. `git rev-parse --is-inside-work-tree` 로 git 저장소 여부 확인
2. **미초기화 시 (자동 수행)**:
   - `git init -b main`
   - 프로젝트 유형에 맞는 `.gitignore` 생성
   - `.github/ISSUE_TEMPLATE/`, `.github/pull_request_template.md` 생성
   - `git add` 후 첫 커밋 (`chore(setup): 프로젝트 초기화`)
   - `git checkout -b develop`
3. **이미 초기화된 경우**:
   - `develop` 브랜치 존재 확인 → 없으면 `git checkout -b develop` (기본 브랜치 기준)
   - develop 체크아웃
4. 이 단계에서는 사용자 확인을 요구하지 않는다. 오류만 보고한다.

## Phase 2: 프로젝트 초기화

1. 계획서(`docs/plans/{프로젝트명}-plan.md`)에서 도메인 목록 추출
2. GitHub Project 칸반 보드 생성 (백로그/진행중/리뷰중/완료)
3. 라벨 생성:
   - 도메인: `domain:{도메인}` (계획서 도메인별)
   - 유형: `type:feature`, `type:bug`, `type:refactor`, `type:docs`, `type:test`
   - 우선순위: `priority:critical/high/medium/low`
4. 계획서의 작업 분해 기반 GitHub 이슈 생성:
   - 각 이슈: 제목, 라벨, 수용 기준, 세부 작업
   - 이슈를 프로젝트 보드에 추가
5. 생성된 이슈 번호를 계획서에 반영 (업데이트)

## Phase 3: 개발 실행

1. 계획서에서 우선순위/의존성 순서로 태스크 정렬
2. 도메인별로 feature 브랜치 생성 및 체크아웃:
   ```bash
   git checkout -b feature/{도메인}/{기능} develop
   ```
3. 도메인별로 `/develop {METHODOLOGY_FLAGS} {태스크}` 실행:
   - 방법론 플래그는 `/start` 인자 그대로 전달
   - `/develop`은 `docs/plans/{프로젝트명}-plan.md` 계획서를 자동 참조
   - `/develop`의 Phase 1(계획)을 건너뛰고 Phase 2(문서화)부터 시작
   - 이슈 번호를 커밋에 자동 연결 (`Refs #{이슈번호}`)
4. 각 도메인 완료 후:
   - 이슈 상태를 "리뷰중"으로 변경
   - 다음 도메인의 feature 브랜치로 이동
5. 모든 도메인 완료 후 Phase 4로

## 오류 처리
- **에이전트 소집 실패**: 해당 관점은 메인 세션이 직접 수행
- **브레인스토밍 교착**: 2라운드까지만 진행 후 메인 세션이 최종 통합
- **사용자 수정 요청 3회 초과**: 계획을 직접 편집하도록 안내
- **`gh` CLI 미인증**: `gh auth login` 안내 후 대기
- **이슈 생성 실패**: 로컬 계획 파일만으로 Phase 3 진행

- 방법론 플래그는 `/start` → `/develop`로 그대로 전달
- 브레인스토밍 중간 산출물은 `docs/plans/`에 보존 (감사 추적용)
